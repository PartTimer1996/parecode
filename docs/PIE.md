# PareCode Project Intelligence Engine (PIE)

**Goal:** Token cost goes down with every session. The scaffold investigates; the model reasons.

**The bet:** Every other agent rebuilds project understanding from scratch on every task. PIE persists and compounds it.

**The philosophy:** The project graph is not a plan-mode feature — it is the underlying model of the project that underpins *every* agent interaction. One-shot tasks, interactive sessions, and plan execution all benefit from knowing what exists and where. PIE is the foundation, not an add-on.

---

## What We're Building (and What We're Not)

**In scope:**
- Persistent symbol graph (survives across sessions)
- Universal graph injection — all agent calls (one-shot, interactive, planning) receive project context
- Directory-based cluster grouping with natural-language summaries
- Task memory (what files were touched, what worked)
- Context weights (stop including things that keep getting wasted)

**Out of scope (for now):**
- tree-sitter — the existing regex extractor in `index.rs` is sufficient
- Louvain/community detection — directory-only clustering is enough for Phase 1
- Cross-import merging — deferred; `index.rs` doesn't parse imports yet, adding it would bloat Phase 1
- Embedding/RAG — structural graph captures exact relationships for free
- Sophisticated "what did the model use" analysis — file overlap heuristic is good enough
- Late symbol binding in execution — `PlanStep.files` stays as resolved paths; symbol-name resolution deferred to Phase 3 or later
- Graph injection into executor steps — executor already has explicit files; project map adds noise not signal

---

## Architecture

Three layers, each with a different timescale:

```
Layer 1: Structural Graph      — rebuilt on file change (deterministic, no model calls)
Layer 2: Project Narrative     — generated once, patched after tasks (~one cheap model call)
Layer 3: Task Memory           — append-only log, grows every session
                    │
                    ▼
         Context Assembler     — queries all three, builds fixed-budget context package
                    │
                    ▼
              Model             — receives pre-assembled context, no exploration needed
```

**Injection scope (Phase 1.5 onwards):**

```
                    ProjectGraph (loaded once at startup, stored on AppState)
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
    One-shot mode   Interactive      Planning
    (system prompt) (system prompt)  (planning prompt)

    Executor steps: EXCLUDED — already have explicit file context
```

**Storage:** `.parecode/` directory (already established)

## Data Structures

### Layer 1 — Structural Graph

**Already implemented in `src/pie.rs`.** Replaces the ephemeral `SymbolIndex` (which was rebuilt fresh on every `/plan`).

```rust
// Persisted to .parecode/project.graph (serde_json)
// JSON chosen over bincode: survives field additions with #[serde(default)],
// human-readable for debugging, no schema-fragility risk during iteration.
struct ProjectGraph {
    schema_version: u32,           // bump when format changes — triggers silent full rebuild
    file_hashes: HashMap<String, String>,  // path → git blob hash (or mtime string fallback)
    file_lines: HashMap<String, usize>,    // path → line count
    symbols: Vec<Symbol>,          // existing Symbol struct, now serde-derived
    by_name: HashMap<String, Vec<String>>,
    clusters: Vec<Cluster>,
    last_indexed: i64,             // Unix timestamp (i64, not SystemTime — easier to serialize)
}

struct Cluster {
    name: String,               // derived from directory name: "tui", "auth", "plan"
    files: Vec<String>,         // relative paths in this cluster
    entry_files: Vec<String>,   // largest/most-symbol-dense files (top 3) — proxy for "important"
    summary: Option<String>,    // generated by model once, ~30 words (Phase 2)
}
```

**Change detection — use batch, not per-file:**
```bash
# ONE subprocess call gives all blob hashes for tracked files
git ls-files -s
# Output: <mode> <hash> <stage>\t<path>
# Parse this once, compare against stored hashes, reindex only what changed
# Fallback (no git): stat() each file, store mtime as string — one syscall per file, no subprocess
```
Never call `git hash-object <file>` in a loop — on a 10K-file monorepo that's 10K subprocesses. `git ls-files -s` is one call regardless of project size.

**Clustering — directory-only:**

Group files by their top-level source directory: `src/tui/*.rs` → cluster `tui`, `src/plan.rs` → cluster `src` (flat files), etc. No import analysis needed. No merging. Simple, predictable, matches the developer's own mental model.

`entry_files` for a cluster = the 3 files with the most symbol definitions (already computed during indexing). Good proxy for "most important" without needing import data.

**Symbol name disambiguation:**

The `by_name` map already stores `HashMap<String, Vec<String>>` — a name can map to multiple files. The `read_symbol` planner tool (Phase 2) must handle this:

```rust
fn read_symbol(name: &str, graph: &ProjectGraph) -> SymbolReadResult {
    let files = graph.by_name.get(name);
    match files.map(|v| v.len()) {
        None => SymbolReadResult::NotFound,
        Some(1) => SymbolReadResult::Code(excerpt_symbol(name, &files[0])),
        Some(_) => {
            // Ambiguous — return structural maps for ALL matches, cost zero reads
            // Model picks the right one, then issues another read_symbol("Name", "path/hint")
            SymbolReadResult::Ambiguous(
                files.iter().map(|f| structural_map_for_symbol(name, f)).collect()
            )
        }
    }
}
```

Ambiguous results count as zero reads against the budget — the model is just orienting. The follow-up targeted read costs one slot.

### Layer 2 — Project Narrative

```rust
// Persisted to .parecode/narrative.json
struct ProjectNarrative {
    schema_version: u32,
    architecture_summary: String,   // ~100 words, one model call on init
    cluster_summaries: HashMap<String, String>,  // cluster name → ~30 word summary
    conventions: Vec<String>,       // ["Services return Result<T>", "Tests in src/__tests__/"]
    last_synthesized: i64,          // Unix timestamp
    patches: Vec<String>,           // ["Added CartService discount logic", ...]
}
```

Conventions are detected zero-cost from the graph:
- Most common return type pattern in `*.service.*` files → "Services return X"
- Test file coverage (`*.spec.*` / `*_test.*` colocated) → "X% of modules have tests"
- Import patterns → detect framework (Angular, React, Express, Axum, etc.)

When `patches.len() > 10` for a cluster, do one cheap model call to re-synthesize that cluster's summary. Clear its patches.

**Narrative generation is triggered explicitly, not at startup.** First `/plan` after a cold narrative triggers it, or the user can run `/pie init`. This avoids surprise model calls during TUI startup.

### Layer 3 — Task Memory

```rust
// Appended to .parecode/task_memory.jsonl (one JSON object per line)
struct TaskRecord {
    id: String,
    timestamp: i64,
    task: String,                  // first 200 chars of user input
    outcome: String,               // "solved" | "partial" | "failed"
    files_modified: Vec<String>,   // proxy for "files that mattered"
    summary: String,               // 1-2 sentence summary of what was done,
                                   // extracted from final assistant message — zero extra model cost
    tokens_used: u32,
    // Weight adjustment data only — not injected into context:
    files_in_context: Vec<String>, // everything that was loaded
}
```

`summary` is the key field — extracted from the last assistant message at `AgentDone` time, no extra model call. Without it, task memory is just a file list and still requires re-reads to understand what changed. With it, each past task is self-describing.

`files_modified` = files the agent actually wrote/patched during execution. Simple, reliable proxy for "useful context".

### Context Weights

```rust
// Persisted to .parecode/context_weights.json
// file path → weight multiplier, default 1.0, clamped [0.2, 3.0]
struct ContextWeights {
    file_weights: HashMap<String, f32>,
}

// After each task:
// - files_modified: weight += 0.1 (were useful)
// - files_in_context but NOT modified: weight -= 0.05 (probably wasted)
```

**Ranking formula for task memory lookup:** `score = intersection_size × recency_weight` where `recency_weight = 1.0 / (days_ago + 1)`. Intersection = count of files in common between the current task's candidate files and the historical task's `files_modified`. This favours specific historical tasks over general ones while ensuring recency breaks ties.

**Weight ceiling note:** A file modified 20+ times hits the 3.0 ceiling. Each subsequent touch beyond that is silently ignored. Files renamed/moved lose their weight — acceptable cost for simplicity; weights rebuild quickly.

## Context Assembler

Replaces ad-hoc context building in `plan.rs` and `agent.rs`. Called before every agent interaction. Zero model calls.

**Budget:** `~1500 tokens` for planning context (configurable). `~600 tokens` for interactive/one-shot context (smaller — the user's task provides more signal).

**Assembly order (fill budget in priority order):**

```
1. Architecture summary          ~100 tokens  always included (Phase 2+)
2. Relevant cluster summaries    ~30 tokens each, 1-3 clusters
3. Relevant conventions          ~20 tokens each, top 3
4. Similar past tasks            ~50 tokens each, top 3 by intersection_size × recency
5. Signal-driven code excerpts   ~200-800 tokens, from stack traces / test failures / file refs
6. Dependency expansion          callees of traced symbols, 1 level deep, if budget remains
7. Fallback: cluster entry files if nothing else matched
```

Phase 1.5 uses a simplified version: just the cluster-grouped symbol section from `to_prompt_section()`. The full assembler is Phase 3.

**Signal parsing** (already partially exists in the agent/hook pipeline):
- Stack trace with `file:line` → read 15 lines around that point
- Test failure with location → read around failure
- Compiler error with location → read around error
- File reference in user input → include structural map for that file (free, no code)
- Pure description → run tests/compiler first (reuse existing hooks), parse output for signals

**Keyword grep anchoring (critical for large codebases):**

For vague tasks with no concrete signals ("the payment flow is broken"), keyword matching against cluster names alone is weak — "payment" might not appear anywhere in the directory structure. Before cluster matching, run a fast grep over the project:

```rust
fn anchor_clusters_by_keyword(task: &str, graph: &ProjectGraph) -> Vec<String> {
    // Extract meaningful words from task (skip stop words)
    let keywords = extract_keywords(task); // ["payment", "flow", "broken"]

    // grep -rl over project source, zero model calls
    let matching_files = keywords.iter()
        .flat_map(|kw| grep_files(kw, &graph.root))
        .collect::<HashSet<_>>();

    // Map matching files → clusters
    graph.clusters_for_files(&matching_files)
}
```

This turns "payment flow is broken" into actual file hits, which anchor the right clusters even when directory names don't match the user's vocabulary. Fast (`grep -l` is cheap), no model calls, high precision on large codebases.

**Relevance:** Before including a code excerpt, multiply its base relevance by the file's context weight. Skip if weight-adjusted relevance ranks it below budget cutoff.

## Planning Phase Changes

Current `plan.rs` gives the model a flat symbol list + the files it asks for. With PIE:

1. Context assembler runs first, produces a `ContextPackage`
2. `ContextPackage` renders into the planning prompt instead of the current symbol index section
3. Planner gets two tools instead of general file access:
   - `list_symbols(file)` — returns structural map, **free** (just graph data)
   - `read_symbol(name)` — returns actual code for a named symbol, **costs a read budget slot**
4. Read budget scales with signal strength:
   - Strong signals (test failure with location): 0 discovery reads needed
   - Medium signals (file reference, vague error): 2 reads
   - Weak signals (pure description): 4 reads

**Important:** Adding planner tools makes `generate_plan()` multi-turn. This is the most significant rewrite in Phase 2 — `plan.rs` currently makes one model call with `&[]` tools. The multi-turn planning loop needs to be built and validated before the rest of Phase 2 is layered on top.

This forces surgical exploration instead of reading whole files.

## Execution Phase Changes

Executor steps deliberately receive **no** project graph injection — the executor already has explicit file context from the step definition. Injecting the project map would add ~600 tokens of noise with no benefit.

After each step completes:
- Reindex modified files (incremental, ~10ms)
- Append to task memory (Phase 3)

## Post-Task Learning

After every completed task (success or failure):

```rust
fn post_task_update(graph: &mut ProjectGraph, narrative: &mut ProjectNarrative,
                    memory: &mut TaskMemory, weights: &mut ContextWeights,
                    task: &CompletedTask) {
    // 1. Reindex modified files
    graph.reindex_files(&task.files_modified);

    // 2. Narrative patch
    let cluster = graph.cluster_for_files(&task.files_modified);
    narrative.patches.push(format!("{}: {}", cluster, task.description_summary()));
    if narrative.patches_for(&cluster).len() > 10 {
        narrative.resynthesize_cluster(&cluster, &model); // one cheap call
    }

    // 3. Task memory append
    memory.append(task.to_record());

    // 4. Weight adjustment
    for file in &task.files_modified {
        weights.increase(file, 0.1);
    }
    for file in task.files_in_context_but_not_modified() {
        weights.decrease(file, 0.05);
    }

    // 5. Persist everything
    graph.save(); narrative.save(); weights.save();
    // task_memory.jsonl is append-only, already written
}
```

## File Layout

```
.parecode/
├── project.graph           # JSON — ProjectGraph (human-readable, schema-safe)
├── narrative.json          # ProjectNarrative (human-readable, editable)
├── task_memory.jsonl       # append-only task log — the only file with accumulated value
├── context_weights.json    # learned file relevance
├── plans/                  # existing plan persistence, unchanged
└── telemetry.jsonl         # existing, unchanged (task_memory subsumes it eventually)
```

**`.gitignore` guidance:**
- `task_memory.jsonl` — **commit this**. It's the moat: cross-device session learning.
- `narrative.json` — **commit this**. Cheap to regenerate but worth preserving across machines.
- `project.graph` — **gitignore this**. Fully reproducible from source, large, changes on every file edit.
- `context_weights.json` — your call. Committing it shares learned weights across machines; ignoring it means each machine learns independently.

Suggested `.gitignore` addition (written to `.gitignore` automatically on first run if not present):
```
.parecode/project.graph
```

All other `.parecode/` files are derived/learnable. If corrupt: delete and rebuild.

### ✅ Phase 1.5 — Universal Graph Injection (complete)

**Goal:** Make the project graph the foundation for *all* agent interactions, not just planning. One-shot tasks and interactive sessions get the same structural orientation that planning already has. Every user gets a fully-indexed project from their very first interaction.

**Rationale:** The graph is built and stored on `AppState`. Warm load is ~10ms; even a cold build on a typical project is 50-150ms — well within the existing 1400ms splash screen window. There is no good reason to withhold project context from the agent on any interaction.

**What to build:**

1. **Build or load graph during the splash screen**
   - The TUI already shows a logo splash for ~1400ms before entering the event loop (`draw_splash()` + `sleep(1400ms)` in `mod.rs`).
   - `draw_splash()` accepts a status string. Change the signature to `draw_splash(f, status: &str)` — replaces the current hardcoded `"loading…"` hint line.
   - Before the splash sleep: spawn `ProjectGraph::load_or_build()` as a blocking task (`tokio::task::spawn_blocking`). While it runs, show `"◈ Setting up Project Intelligence…"` in the hint line on a cold build, `"◈ loading project graph…"` on warm.
   - Await the result before exiting the splash. Store as `AppState.project_graph: Option<ProjectGraph>`.
   - On completion, the status line already transitions to `◈ PIE  N clusters · M files · warm/indexed`.
   - The cold build is invisible to the user — it happens behind a splash they're already watching.

2. **Inject graph into the system prompt for all agent calls**
   - Add a section to the system prompt assembly: if `state.project_graph.is_some()`, call `graph.to_prompt_section(8)` and append it.
   - This fires for: normal interactive tasks, one-shot CLI mode (where a graph exists on disk), plan generation (already covered — no change needed there).
   - **Explicitly skip** for executor step calls — the executor system prompt is built separately (QW3); injecting the full project map there adds noise, not signal.

3. **Graph refresh after task completion**
   - After any agent run that modifies files, call `graph.incremental_update()` and `graph.save()`.
   - This keeps the in-memory graph current across tasks in the same session without a full rebuild.
   - Tie into the existing `AgentDone` event handling.

4. **One-shot mode**
   - In `run_single_shot()` / `run_single_shot_quick()` in `main.rs`, call `ProjectGraph::load_or_build()` before building the agent config — always, warm or cold.
   - On a cold build, print `"  ◈ indexing project…"` to stdout before it starts. The 50-150ms is invisible against the first model roundtrip.
   - Inject `graph.to_prompt_section(8)` into `AgentConfig.project_context`. The agent receives the same orientation as interactive mode.

**Token cost of injection:** `to_prompt_section(8)` on a typical project produces ~500-800 tokens. This is always cheaper than one speculative file read (~200-1500 tokens per file). The break-even is immediate: if it prevents a single unnecessary read, it paid for itself.

**What does NOT change:**
- Executor step calls are excluded. They already have explicit `step.files` context.
- The existing warm status line (`◈ PIE  N clusters · M files · warm`) is unchanged.
- `generate_and_show_plan()` can drop its own `load_or_build()` call — the graph is already on `AppState`.

**Validation:**
- First run (cold): splash shows `"◈ Setting up Project Intelligence…"`, then transitions normally. No hang, no blank screen.
- Subsequent runs (warm): splash shows `"◈ loading project graph…"`, graph loads in <10ms.
- Run a one-shot task on a project with an existing graph. Confirm the agent goes directly to the right files without exploratory reads.
- Confirm executor steps do NOT receive the project map (check system prompt token count before/after a plan execution step).
- Token count for a typical interactive task should drop by ~10-20% for projects with 10+ files due to eliminated orientation reads.

## Known Gotchas

**Ambiguous symbol names** — handled by `SymbolReadResult::Ambiguous`, documented in the data structures section above. The single most common surprise in monorepos.

**`git ls-files -s` vs per-file hash** — one subprocess vs N subprocesses. Non-negotiable for large projects. The per-file approach will work fine on 50-file projects and silently destroy startup time on 5,000-file ones. Always implement the `mtime` fallback for non-git repos.

**Keyword grep for vague tasks** — without this, PIE's cluster matching fails silently on large codebases when the user's vocabulary doesn't match directory names. "The payment flow" won't match a directory called `src/billing/`. The grep anchor solves this at near-zero cost.

**Narrative staleness** — the architecture summary is generated once and patched incrementally. If a project undergoes major restructuring (new framework, major refactor), the patches approach won't catch it. The `/pie reset` command is the escape hatch. Make it visible in the stats bar when the narrative is older than 30 days or has >20 accumulated patches.

**Context weights decay too slowly vs too aggressively** — `+0.1 / -0.05` are starting values. A file that's been wasted 20 times in a row hits the 0.2 floor. Watch the actual weight distributions after phase 3 ships and tune. The clamp `[0.2, 3.0]` prevents any file from being permanently excluded or dominating.

**Multi-turn planning loop** — `generate_plan()` currently makes one model call. Phase 2 changes this to a loop. Build and validate the loop standalone before integrating context assembler on top of it. It's the highest-risk rewrite in the whole project.

**Planner tool call that triggers a model call (narrative generation)** — the first `/plan` on a cold narrative will make two model calls: one for narrative, one for the plan itself. This is fine but should be communicated to the user ("generating project narrative...") to avoid the appearance of a hang.

**Universal injection token budget** — injecting `to_prompt_section(8)` adds ~500-800 tokens to every agent call. For a 10-message session on a small task, that's ~5-8K extra tokens carried across all turns in the system prompt. This is already cheaper than one speculative file read, but monitor whether `max_clusters` should be tuned lower (4-5) for interactive mode vs planning mode (8). Small tasks warrant a smaller map.

**Multi-project / subdirectory confusion** — `.parecode/project.graph` is relative to cwd. A user working in a monorepo subdirectory gets a different graph than one at the root. This is intentional: the graph scopes to where you run PareCode.

**Splash duration vs cold build time** — the existing splash is 1400ms. A cold build on a large project (500+ files) could approach 300-500ms, which fits comfortably. If a project is pathologically large (10K+ files) the build could exceed the splash window — in that case the splash simply holds until the build finishes, which is fine. The user sees a message explaining what's happening. Don't cap the build artificially; let it complete.

**prompt caching (Anthropic/OpenAI)** — as provider-side caching improves, some of the token cost argument weakens for those providers. The learning loop argument doesn't weaken — it's about *what* gets sent, not what it costs per token. A tool that learns your project's patterns and pre-loads the exact files needed — even when using a fully cached provider — is more accurate and faster than one that re-explores from scratch. Lead with quality, use token savings as supporting evidence.

## Key Design Decisions

**Why serde_json not bincode for project.graph?**
Bincode is schema-fragile: adding a struct field breaks deserialization even with `schema_version` guards, unless every field is carefully wrapped. During rapid iteration, that's a constant friction. JSON with `#[serde(default)]` survives any field addition or removal. Parse time is not a bottleneck at the scale of a typical codebase (< 5ms to deserialize a 500-file project graph). Switch to bincode if profiling shows it matters.

**Why not tree-sitter yet?**
The regex extractor in `index.rs` already covers the essential structural data for planning. Tree-sitter adds native C compilation, binary size, and per-language grammar crates. The value of PIE is the *persistence and learning* architecture — not extraction accuracy. Upgrade the extractor after the learning loop is proven.

**Why directory clustering over community detection?**
Directory structure is the developer's own mental model of the codebase. Community detection might find "better" clusters by import density, but developers expect "authentication stuff" to be the auth cluster, not whatever the algorithm decided. No import parsing needed in Phase 1 — `entry_files` by symbol count is a good enough proxy for "important files in this cluster".

**Why append-only task memory?**
Simple, safe, git-friendly, greppable. No database. Compaction handles growth. The `task_memory.jsonl` format is readable by any tool and survives schema changes (just add fields, old records have defaults).

**Why file-modification as the "useful context" proxy?**
Parsing the model's reasoning to detect which symbols it referenced is fragile. Files that were modified were definitely used. Files in context that weren't touched were probably wasted. This heuristic is noisy but directionally correct and gets better as the task history grows.

**Why is `TaskRecord.summary` the most important field?**
Without it, task memory is a file list — useful for weight adjustment but not for context injection. The receiving model still has to re-read those files to understand what changed. A 1-2 sentence summary extracted from the final assistant message at zero cost makes each past task self-describing: "Updated multi-turn loop signature — narrative param added, on_chunk bound tightened." That's the difference between "you probably want to look at plan.rs" and "here's what was last changed there and why." The summary is what turns a lookup table into actual institutional memory.

**Why the learning loop is the defensible moat (not just token savings)?**
Provider-side prompt caching (Anthropic, OpenAI) will reduce the raw token cost argument over time for cloud providers. The learning loop argument is orthogonal to caching: it's about sending the *right* context, not sending *less* context. A tool that learns your project's patterns and pre-loads the exact files needed — even when using a fully cached provider — is more accurate and faster than one that re-explores from scratch. Lead with quality, use token savings as supporting evidence.

**Why exclude executor steps from graph injection?**
Executor steps already have explicit, pre-resolved file context from the plan. The executor's job is surgical: edit these specific files according to this specific instruction. Injecting the full project map adds noise, increases context size, and could distract the model from its narrow task. The planner (which does need the full picture) already gets the graph. The executor should not.

**Why not build the graph cold at startup?**
Zero-latency startup is a feature. A cold build on a large project could take 200-500ms, which is a noticeable delay before the TUI appears. The tradeoff is acceptable: the graph builds on first `/plan`, and every session after that benefits from the warm load.
